系统设计 叙事引擎 JSON数据驱动

时间当铺 · 系统设计文档

这个游戏最难的地方不是叙事，是怎么让叙事可以被复用。
讲讲我具体是怎么想、怎么做的。

🎮 先去玩一下

一开始我想解决的问题

玩叙事解谜的时候，通关一次就不想再打开了——剧情已经知道了，再玩一遍没有意义。

但如果我告诉你，你上一次的某个选择，改变了另一个人几十年后的人生呢？

那种感觉就不一样了。不是"我读过这个故事"，而是"我的选择真的留下了什么"。这个游戏想做的，就是这种感觉。

核心设计目标

  • 叙事内容不因单次游玩耗尽——至少有 8种结局，隐藏内容覆盖率 ≥40%
  • 蝴蝶效应可被感知——玩家能追溯到"是哪个选择造成了什么改变"
  • 多周目有递进感——每次通关解锁新内容，不是重复劳动

具体怎么做

1. 全部内容用 JSON 定义

这样做的好处是：策划想改对话、想加选项、想调结局触发条件，不需要动代码。程序只负责读取 JSON、渲染、响应事件。具体内容完全由策划控制。

// 一个选择节点的定义

{
  "id": "ch1_choose_01",
  "npc": "林知予",
  "text": "……这个选择，我只在档案最深处见过一次。",
  "options": [
    { "text": "我不在乎，继续", "next": "ch1_ending_dark", "ripple": ["ripple_doom_01"] },
    { "text": "有没有别的办法？", "next": "ch1_secret_path", "ripple": [] }
  ]
}

2. 蝴蝶效应的实现逻辑

每个选择可以附加"涟漪"。涟漪就是一个字符串ID，比如 ripple_01，记录到玩家的存档里。

后续章节里，某些对话会有"触发条件"——当玩家存档里包含某个涟漪ID时，触发不同的对话分支。

举个例子：

Ch1 选择放弃某段记忆 → 产生涟漪 ripple_forgotten_01
Ch4 进入1962年时间线 → NPC"陈守"会检查你是否持有这个涟漪
如果有，他会多说一句："那封信……是你寄的吗？"
如果没有，他会说另一套话
→ 玩家意识到"我的上一个选择，改变了这里发生的事"

3. 双层存档：当前周目 + 跨周目

我设了两层存档：NormalSave（当前这一局）和 MetaSave（所有周目累计的数据）。NormalSave 每次重开就清空，但 MetaSave 一直保留。这样第二次打开游戏，系统会知道你是"回来过的人"，解锁一些普通玩家看不到的内容。

比如林知予会说："你又来了。上一次，你在这里选了另一条路。"——这句话普通玩家第一次看不到，只有回到这个游戏的玩家才会触发。这种设计让"回到游戏"本身变成一种奖励。

系统之间的关系

说白了，这个游戏就四个东西在互相配合：

JSON配置层

所有对话、选项、涟漪定义在这里，是内容本身

叙事引擎

读取配置、控制流程、渲染画面，只负责执行不负责内容

涟漪系统

记录玩家的选择结果、查询涟漪状态、判断触发条件

存档系统

双层存档读写，NormalSave 和 MetaSave 各司其职

数据流向是单向的：JSON → 叙事引擎 → 涟漪系统（记录） → 存档系统（持久化） → 下一次读取。程序不直接修改 JSON，策划改了 JSON 也不需要通知程序。

做完之后我觉得还能往哪走

v1.1 · 涟漪日志

很多玩家不一定能感知到自己产生了涟漪。做一个"涟漪日志"界面，让他们能随时查看自己积累了什么、这些涟漪目前影响到了哪些地方，降低认知门槛。

v1.2 · 新章节"守钟人"

引入一个新角色，和林知予形成镜像对照——她是被困在过去的人，他是被困在未来的人。这个章节独立于主剧情，但可以复用同一套涟漪系统，用来验证它能不能真的被扩展。

v2.0 · 社区模组支持

如果 JSON Schema 定义得足够清晰，其实可以开放给社区创作者写新章节。我一直想看看"叙事平台"有没有可能——不是做一个游戏，而是做一个可以让人创作叙事游戏的东西。
